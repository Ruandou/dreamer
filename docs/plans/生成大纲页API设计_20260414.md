## 一、概述

**核心设计原则**：服务端维护上下文，API 极简，保证剧集连贯性。

生成大纲页涉及四个核心 API（**剧本**与 **解析骨架** 分属不同端点）：

| API | 方法 | 说明 |
|-----|------|------|
| 生成第一集 | `POST /api/projects/:projectId/episodes/generate-first` | 基于创意生成全剧梗概 + 第一集剧本 |
| 生成剩余所有集 | `POST /api/projects/:projectId/episodes/generate-remaining` | 异步任务，批量生成第 2 集至目标集数；**不**依赖 **`Project.visualStyle`** |
| **解析剧本** | **`POST /api/projects/:projectId/parse`** | 异步：实体提取、形象槽位、分集概要等；**不**在本端点启动分镜/视频全链路 |
| 查询任务进度 | `GET /api/pipeline/job/:jobId`（现状） | 轮询批量剧本、**解析剧本** 等 Job（路径见 §1.1） |

所有上下文（全剧梗概、累积摘要、已生成剧集）由服务端从数据库读取并拼接，前端只需传递最小必要参数。

### 1.1 与仓库现状的差异（实现前必读）

| 设计项 | 说明 |
|--------|------|
| **进度查询路径** | 设计稿若写 `GET /api/pipeline-jobs/:jobId`，当前仓库为 **`GET /api/pipeline/job/:jobId`**（前缀 `/api/pipeline`）。新实现可二选一：**沿用现有路径**（推荐）或新增别名路由后废弃旧路径。 |
| **响应包裹** | 现有 Pipeline Job 接口返回 **`{ success, data }`**。本文档示例为扁平 JSON；前后端约定时可统一为扁平或保留 `data` 包裹。 |
| **`Project.storyContext`** | v4.1 摘要文档与当前 Prisma **均无**该字段；实现本设计需 **迁移新增**（建议 `TEXT`，可空）。 |
| **`PipelineJob.type`** | 当前 `PipelineJob` **无** `type` 列；若需区分 `script-batch` 与主流水线，需 **迁移新增**或使用 `currentStep` + 约定枚举。 |
| **路由注册** | `generate-first` / `generate-remaining` 须注册在 **`/api/episodes/:id` 动态路由之前**，或与 `projectId` 绑在 `projects` 子路由下，避免被 `/:id` 吞掉。 |

## 二、API 详细设计

### 2.1 生成第一集

**端点**

```http
POST /api/projects/:projectId/episodes/generate-first
```

**请求体**

```json
{
  "description": "现代工程师宋应星意外穿越到古代，附身在一个穷困书生身上，但他发现随身携带的手机竟然还能连接现代搜索引擎……"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| description | string | 是 | 用户输入的故事创意 |

**响应体**

```json
{
  "episode": {
    "id": "ep_001",
    "episodeNum": 1,
    "title": "苏醒",
    "script": {}
  },
  "synopsis": "宋应星是一名现代工程师，因一场意外穿越到古代..."
}
```

**服务端处理逻辑**

1. 调用 AI（如 DeepSeek）生成：**全剧梗概**、**第一集完整剧本**（`script`，结构与共享类型 `ScriptContent` 等约定一致）。
2. 持久化：
   - `Project.synopsis` = 全剧梗概；
   - 若尚无 `Episode` with `episodeNum=1`，则 **create**；否则 **update**；
   - `Episode.script` 写入第一集剧本；
   - **`Project.storyContext`**（新增字段）：初始化为「梗概 + 第一集摘要」（摘要由 §3.4 提取）。
3. 可选：同步更新 `Project.description` = 请求中的 `description`（与交互稿「创意」字段对齐）。
4. 返回第一集实体及 `synopsis` 字符串。

---

### 2.2 生成剩余所有集（异步）

**端点**

```http
POST /api/projects/:projectId/episodes/generate-remaining
```

**请求体**

```json
{
  "targetEpisodes": 60
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| targetEpisodes | number | 是 | 目标总集数，必须 **大于** 当前已生成集数 |

**响应体**

```json
{
  "jobId": "job_123456",
  "status": "processing",
  "targetEpisodes": 60,
  "generatedCount": 1,
  "message": "任务已启动，正在生成第 2/60 集"
}
```

**服务端处理逻辑**

1. 校验 `targetEpisodes` > 当前 `max(episodeNum)`（或已存在集数）。
2. 创建 **`PipelineJob`**（建议 `type = 'script-batch'` 或等价元数据），`status = pending/processing`，返回 `jobId`。
3. 异步执行批量生成（§3.3）；HTTP 立即返回。
4. 前端使用 `jobId` 轮询 **§2.4**。

**与 `Project.visualStyle`**：**不要求、不校验**。本任务仅依据第一集与 **`storyContext`** 等续写第 2…`targetEpisodes` 集的 **`Episode.script`**。  
**至少一项 `visualStyle`** 的约束**仅**适用于 **§2.3 解析剧本**（制作准备），与批量写集解耦。

---

### 2.3 解析剧本（制作准备，异步）

**端点**

```http
POST /api/projects/:projectId/parse
```

**前置校验（与交互稿一致，服务端须兜底）**

- 项目存在且归属当前用户。
- **第一集** `Episode.script` 已存在（非空）。
- **`Project.visualStyle`** 至少一项。
- 若 **`targetEpisodes`** 下仍有缺集 **`script`**：可在本任务**开始前**先编排 **补全剩余集剧本**（与「稍后再说」语义一致），或作为本 Job 内第一步执行；具体以 executor 实现为准，但须避免无剧本可解析。

**请求体（可选）**

```json
{
  "targetEpisodes": 60
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| targetEpisodes | number | 否 | 缺省则用项目上的目标集数；用于补全缺集时的上界 |

**响应体**

```json
{
  "jobId": "job_parse_001",
  "status": "processing",
  "message": "解析任务已启动"
}
```

**服务端处理逻辑（职责边界）**

1. 创建 **`PipelineJob`**（建议 `type = 'parse-script'` / `skeleton` 或等价元数据），`status = pending/processing`，返回 `jobId`；HTTP 立即返回。
2. 异步执行（顺序可由实现微调，但**不得**在本 Job 内启动「分镜生成 / 视频生成」全链路）：
   - 从已落库的各集 **`script`** 提取并 upsert **`Character`、`Location`**；
   - 为每个 **`Character`** 创建基础 **`CharacterImage`**（`avatarUrl = null`）；
   - 为各 **`Episode`** 写入 **`synopsis`**（若为空）。
3. 前端跳转项目详情后，使用同一 **`GET /api/pipeline/job/:jobId`** 轮询直至 `completed` / `failed`。

**明确不包含**：角色图/场景图生成、**Scene** 落库、分镜、视频——由详情页后续能力触发。

---

### 2.4 查询生成进度

**端点（设计）**

```http
GET /api/pipeline/jobs/:jobId
```

**与现状对齐**：实现阶段优先使用已有 **`GET /api/pipeline/job/:jobId`**；若统一为 `.../jobs/:jobId`，需在 `pipeline.ts` 增加路由并保持鉴权与所有权校验一致。

**响应体（进行中）** — 可与现有 `progress` 字段扩展兼容：

```json
{
  "jobId": "job_123456",
  "status": "processing",
  "progress": {
    "current": 12,
    "total": 60,
    "message": "正在生成第 12/60 集"
  }
}
```

**响应体（已完成）**

```json
{
  "jobId": "job_123456",
  "status": "completed",
  "progress": {
    "current": 60,
    "total": 60
  },
  "result": {
    "episodeIds": ["ep_002", "ep_003"]
  }
}
```

**响应体（失败）**

```json
{
  "jobId": "job_123456",
  "status": "failed",
  "error": "生成第 12 集时超时，请重试"
}
```

> 实现时可将 `progress.current/total/message` 写入 `PipelineJob` 的扩展 JSON 字段，或复用 `progress` 整数 + 单独 `progressMeta` 列，避免破坏现有只存 0–100 的语义。

## 三、上下文管理机制（服务端核心）

### 3.1 数据模型补充

在 **Project** 上新增（相对当前 Prisma）：

| 字段 | 类型 | 说明 |
|------|------|------|
| storyContext | Text? | 累积剧情摘要。初始化：梗概 + 第一集摘要；每生成一集后追加本集摘要。 |

与 v4.1 已存在字段的关系：**不替代** `synopsis`；`synopsis` 保持「全剧梗概」，**storyContext** 用于「前情滚动摘要 + 可截断策略」。

### 3.2 生成单集时的上下文构建（伪代码）

以下为逻辑示意，数据库 API 以 Prisma 为准。

```typescript
/**
 * 为生成第 N 集构建完整的上下文 Prompt
 */
async function buildContextForEpisode(projectId: string, targetEpisodeNum: number) {
  const project = await prisma.project.findUnique({
    where: { id: projectId },
    include: {
      episodes: {
        where: { episodeNum: { lt: targetEpisodeNum } },
        orderBy: { episodeNum: 'asc' }
      }
    }
  })
  if (!project) throw new Error('PROJECT_NOT_FOUND')

  const synopsis = project.synopsis ?? ''
  const accumulatedContext = project.storyContext?.trim() || synopsis

  const recentEpisodes = project.episodes.slice(-3)
  const recentScripts = recentEpisodes
    .map((ep) => `第${ep.episodeNum}集：\n${JSON.stringify(ep.script)}`)
    .join('\n\n')

  return `
你是一个专业的短剧编剧。请根据以下信息，续写第 ${targetEpisodeNum} 集的完整剧本。

【全剧梗概】
${synopsis}

【前情摘要】
${accumulatedContext}

【最近剧集详细内容】
${recentScripts}

请生成第 ${targetEpisodeNum} 集的剧本，要求：
- 保持人物性格和剧情逻辑连贯
- 每集包含 8-15 场戏
- 输出格式为 JSON，与系统约定的 `ScriptContent` / `Episode.script` 结构一致
`
}
```

### 3.3 批量生成流程（伪代码）

```typescript
async function generateRemainingEpisodes(jobId: string, projectId: string, targetEpisodes: number) {
  const project = await prisma.project.findUnique({
    where: { id: projectId },
    include: { episodes: { orderBy: { episodeNum: 'asc' } } }
  })
  if (!project) return

  const maxNum = Math.max(0, ...project.episodes.map((e) => e.episodeNum))
  const startNum = maxNum + 1

  for (let epNum = startNum; epNum <= targetEpisodes; epNum++) {
    const prompt = await buildContextForEpisode(projectId, epNum)
    const scriptJson = await callAI(prompt)

    await prisma.episode.create({
      data: {
        projectId,
        episodeNum: epNum,
        title: (scriptJson as { title?: string }).title ?? `第${epNum}集`,
        script: scriptJson as object
      }
    })

    const newSummary = await extractSummary(scriptJson)
    await prisma.project.update({
      where: { id: projectId },
      data: {
        storyContext: [project.storyContext, newSummary].filter(Boolean).join('\n')
      }
    })

    await updateJobProgress(jobId, epNum, targetEpisodes)
  }
}
```

### 3.4 摘要提取（示意）

```typescript
/** 从剧本 JSON 提取约 200 字关键剧情摘要（AI 或规则） */
async function extractSummary(scriptJson: unknown): Promise<string> {
  const summary = await callAI(`
请用 200 字以内总结以下剧本的关键剧情：
${JSON.stringify(scriptJson)}
`)
  return summary
}
```

## 四、前端集成示例

### 4.1 生成第一集

```javascript
const res = await fetch(`/api/projects/${projectId}/episodes/generate-first`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ description: idea })
})
const data = await res.json()
// 更新页面：显示梗概和第一集剧本
```

### 4.2 生成剩余集并轮询

```javascript
const res = await fetch(`/api/projects/${projectId}/episodes/generate-remaining`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ targetEpisodes: 60 })
})
const { jobId } = await res.json()

const pollInterval = setInterval(async () => {
  const progressRes = await fetch(`/api/pipeline/job/${jobId}`, {
    headers: { Authorization: `Bearer ${token}` }
  })
  const body = await progressRes.json()
  const progress = body.data ?? body

  updateProgressBar(progress.progress?.current, progress.progress?.total)

  if (progress.status === 'completed') {
    clearInterval(pollInterval)
  } else if (progress.status === 'failed') {
    clearInterval(pollInterval)
    showError(progress.error)
  }
}, 2000)
```

（轮询 URL 以最终实现为准；若仍为 `{ success, data }`，取 `body.data`。）

## 五、与生成大纲页交互的对应关系

| 用户操作 | 调用的 API | 请求参数 | 页面响应 |
|----------|------------|----------|----------|
| 页面初始加载（创意已存在） | `generate-first` | description | 展示梗概 + 第一集剧本 |
| 点击「生成剩余所有集」 | `generate-remaining` | targetEpisodes | 按钮变为进度条，开始轮询 |
| 轮询进度 | `GET …/pipeline/job/:jobId` | — | 更新进度条文案 |
| 全部生成完成 | — | — | 展示「全部剧本预览」面板 |

## 六、错误处理与边界情况

| 场景 | HTTP | 响应体 | 处理 |
|------|------|--------|------|
| 创意为空 | 400 | `{ "error": "description is required" }` | 前端提示「请输入故事创意」 |
| targetEpisodes ≤ 已生成集数 | 400 | `{ "error": "targetEpisodes must be greater than existing episode count" }` | 前端修正或提示 |
| 单集 AI 失败 | 500 / 任务 failed | `{ "error": "AI generation failed" }` | 日志 + Job `failed` + `error` 文案 |
| 批量中途失败 | — | `PipelineJob.status = failed` | 前端「生成中断，可重试」 |
| **重试** | 再次 `generate-remaining` | 同 targetEpisodes | 服务端根据已存在 `Episode` + `storyContext` **从断点续写**（`startNum = max(episodeNum)+1`） |

## 七、性能与成本

| 阶段 | Token 量级（估算） | 耗时（估算） | 建议 |
|------|-------------------|--------------|------|
| 第 1 集 | ~2000–3000 | 10–15 s | 可同步等待或短轮询 |
| 第 2–60 集 / 每集 | ~4000–6000（含上下文） | ~10–15 s/集 | 必须异步；总时长约 10–15 min |
| storyContext 增长 | 每集约 +200 字摘要 | 随集数缓慢增长 | 可设上限 + 滑动窗口，避免无限膨胀 |

**优化**：上下文重复部分可配合供应商缓存；摘要提取可用更小模型降低成本。
