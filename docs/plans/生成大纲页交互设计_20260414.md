## 1. 页面定位

| 项 | 说明 |
|----|------|
| **页面路由** | **`/generate?projectId=xxx`**（主路径须带 **`projectId`**，与落地决策一致）。**不**以仅 `jobId` 作为等价主路径。 |
| **核心职责（两步解耦）** | **① 剧本**：梗概 + 第一集预览；可选 **批量生成剩余集** `Episode.rawScript`（**不**依赖 **`Project.visualStyle`**）。**② 制作准备**：用户点击 **「解析剧本」** 前须选择 **`Project.visualStyle`**（至少一项）——从已生成剧本中提取 **Character / Location**、创建 **CharacterImage** 基础槽位、补全 **`Episode.synopsis`** 等；**不**在本按钮内触发生图、分镜、视频。 |
| **不在本页完成** | 场次（Scene）级精修、形象图/场景图生成、分镜、视频、合成导出（在 **项目详情页** 及后续任务中完成） |

### 1.1 与 v4.1 数据模型的对齐要点

- **Project**：`name`、`description`（故事创意可映射于此或扩展草稿字段）、`synopsis`（故事梗概）、`visualStyle: String[]`（本页多选写入）。
- **Episode**：`episodeNum`、`title`、`synopsis`、`rawScript`（Prisma 字段名 `rawScript`，列映射为 `script`）；**无** `sceneIndices` 持久化（v4.1）。
- **第一集预览**：展示 **Episode 1 的 `rawScript`（JSON）** 渲染为可读「Scene 段落」；高级编辑直接编辑该 JSON，与「场次管理」中逐 Scene 编辑互补。
- **流水线 / 异步任务**：批量剧本生成、**「解析剧本」** 等后台任务均落 **`PipelineJob` + `PipelineStepResult`**（或等价 Job 表）；进度可轮询 **`GET /api/pipeline/job/:jobId`**，与现有异步模式一致。

### 1.2 主按钮命名

推荐使用 **「解析剧本」**：简洁、直观，表达「AI 分析已写剧本并提取制作所需结构化信息」；避免使用「进入流水线」等易与「全自动跑分镜/视频」混淆的文案。内部实现可与「骨架搭建 / 实体提取」概念对应。

## 3. 区域详解

### 3.1 页面头部、故事创意区、故事梗概区、视觉风格区

- **头部**：左侧返回项目列表；标题「生成大纲」；右侧「保存草稿」——将当前 `Project`（`description` / `synopsis` / `visualStyle`）及已落库的 `Episode`（至少第一集 `rawScript`）持久化；若有未保存变更，点「返回」时二次确认。
- **故事创意**：对应用户输入的**一句话/一段话创意**；建议映射 **`Project.description`**（或后续「草稿 JSON」字段，MVP 可用 `description`）。
- **故事梗概**：对应 **`Project.synopsis`**；与创意区并列，支持编辑与「重新生成」。**注意**：「重新生成」梗概若会联动覆盖第一集，需在 UI 上明确二次确认（与实现一致）。
- **视觉风格**：**在点击「解析剧本」前至少选一项**（解析与后续制作准备需要）；**不要求**在「生成剩余 N 集」之前选择。多选结果写入 **`Project.visualStyle`**（`String[]`），可约定**选择后即时保存**（`PUT`）或随「保存草稿」一并提交。选项集由产品常量表维护；与后续 `Scene.visualStyle` 继承策略在实现文档中约定（v4.1）。

### 3.2 第一集剧本预览区

| 元素 | 交互 | 说明 |
|------|------|------|
| 默认展示 | 仅展示 **第一集** `rawScript` 解析后的**前 2 个文学 Scene**（非 DB `Scene` 实体；v4.1 中 **Scene** 为流水线后续落库单位） | 渲染规则：从 `ScriptContent` / 共享类型约定的 JSON 结构中取场次级描述与对白 |
| **[展开]** | 展开第一集全部「剧本 Scene」段落 | 折叠态显示总场次数 |
| **[✏️ 高级编辑]** | 弹层内嵌 **Monaco Editor**，编辑 **Episode(episodeNum=1) 的 `rawScript` JSON** | 保存走 `PATCH/PUT` 更新 Episode；需 JSON 校验与体积提示 |

### 3.3 剧本生成操作区（核心）

| 元素 | 交互 | 说明 |
|------|------|------|
| **总集数** | 静态文案，如「总集数设定：**60** 集」 | MVP 默认 **60**；数值来源：**项目创建参数**或 **Pipeline 入参 `targetEpisodes`**（与 `POST /api/pipeline/execute` 已有字段对齐）；V2 可在项目设置中可配置 |
| **[📖 生成剩余 N 集剧本]** | 主按钮，`N = 总集数 - 1`；点击后异步批量生成第 2…`总集数` 集的 **`Episode.rawScript`** | **不**校验、**不**依赖 **`Project.visualStyle`**（与「解析剧本」解耦）。后端创建独立 **`PipelineJob`**（建议 `jobType`：`script-batch`），前端轮询 **`GET /api/pipeline/job/:jobId`**；按钮 loading + 进度文案 |
| **[⏭️ 稍后再说]** | 次按钮；本页**不**触发批量写集 | 用户可先打磨仅含第一集的剧本；后续点击 **「解析剧本」** 时，若仍缺第 2…`targetEpisodes` 集，**由服务端在解析任务前或解析流水线内补全**（与 API/executor 约定一致，避免用户被阻塞） |

**进度文案示例**：「正在生成第 5/60 集剧本，预计剩余约 X 分钟」——由服务端根据已写入集数 / ETA 估算（可选）。

### 3.4 全部剧本预览区（批量生成完成后）

当「生成剩余 N 集」**成功完成后**，在剧本操作区位置**自动出现**（或替换为）下列结构：

│  │ ...（该集 rawScript 渲染结果，只读）                                      │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                                                                              │

- **集数选择器**：下拉或上/下一集；数据来自 **`Episode` 按 `episodeNum` 排序**。
- **内容**：只读渲染各集 `rawScript`；**不在本页做逐字编辑**（避免与 Monaco 全量 JSON 冲突）；精修引导至场次管理。

### 3.5 底部操作

| 按钮 | 行为 |
|------|------|
| **返回修改** | 返回 **`/projects`**；若有未保存变更 → **确认弹窗** |
| **🔍 解析剧本** | 见 **§3.6**（原「进入流水线」已废止为该动作） |

### 3.6 「解析剧本」按钮（主 CTA）

**校验（前端 + 后端）**

- 至少 **第一集** `Episode.rawScript` 已存在（可展示为已生成）。
- **`Project.visualStyle`** 至少 **一项**（与 §3.1 一致）。

**提交**

- 调用 **`POST /api/projects/:projectId/parse`**（路径以实现为准；可与现有路由前缀对齐）。
- 服务端创建 **`PipelineJob`**（或专用任务记录，见 API 设计扩展），执行 **后台任务**，HTTP 立即返回 **`jobId`**（若与解析任务一一对应）。

**跳转**

- **`/project/:id`**（项目详情页）；详情页 **顶部** 展示解析进度（如「正在解析剧本，提取角色和场景…」），与轮询或 WebSocket 约定一致。

**后台任务职责（本按钮范围内）**

1. **实体提取**：从**已存在**的各集 **`rawScript`** 中识别并创建 / 更新 **`Character`、`Location`**（与 v4.1 一致；`Location.name` 映射见 schema `@map("location")`）。
2. **形象槽位**：为每个 **`Character`** 创建一条基础 **`CharacterImage`**（`avatarUrl = null`，类型等字段按现有约定）。
3. **分集概要**：为各 **`Episode`** 填充 **`synopsis`**（若此前未生成或为空则写入）。

**明确不包含**：生成角色图/场景图、创建 **Scene**、分镜、视频、合成——均在详情页后续步骤触发。

### 3.7 与任务清单的对应（产品拆分）

| 序号 | 任务内容 | 触发方式 |
|------|----------|----------|
| 3 | 生成第一集剧本 | 进入页面自动触发（或首屏骨架屏后触发） |
| 6 | 生成剩余所有集剧本 | 用户点击「生成剩余 N 集」 |
| 4 | 实体提取（角色、场景） | 用户点击「解析剧本」 |
| 5 | 创建形象槽位 | 用户点击「解析剧本」 |

**设计意图**：**写剧本**与**解析骨架**为两个独立用户动作；用户可先只保留第一集并反复打磨，满意后再解析。

### 3.8 项目详情页承接（概要）

用户自本页跳转 **`/project/:id`** 后：

- **顶部**：解析进度条/文案直至任务结束。
- **左侧 Tab**（角色库、场地库、分集管理等）：解析进行中可 **骨架屏/占位**；完成后 **自动刷新** 展示卡片与列表。
- **后续**：形象图、场景图、场次、视频等在详情页内继续（本文不展开）。

## 4. 加载与状态反馈

### 4.1 初始进入

- 若 **第一集 `rawScript` 为空**：自动触发「生成第一集」流程（骨架屏 + 可取消提示）；可与梗概生成并行或串行（产品可配置「先梗概后第一集」）。
- 若 **梗概为空**：同步触发梗概生成，写入 **`Project.synopsis`**。

### 4.2 批量生成剩余集期间

- 主按钮区变为 **进度条 + 文案**（见 §3.3）。
- **允许离开页面**：任务后台继续；再次进入同一路由时，根据 **`projectId` + 未完成 Job** 自动恢复轮询与 UI 状态（需后端可查询「进行中的 script-batch Job」或统一 Pipeline Job 列表过滤）。

### 4.3 批量完成后

- 进度区收起，展示 **§3.4 全部剧本预览**。
- **「保存草稿」**在任意有变更时刻可用；批量完成后可提示用户在 **视觉风格** 区至少选一项，再点击 **「解析剧本」** 进入制作准备（风格**仅**解析步骤需要），或继续在本页编辑剧本。

## 5. 数据与 API 映射

### 5.1 目标契约（实现时可与现有路由合并或别名）

以下路径为**建议 REST 形状**；**现状**列标注仓库内已有接近能力，便于一期对齐命名与减少重复端点。

| 操作 | 建议 API | 请求体 / 说明 | 现状（2026-04） |
|------|-----------|----------------|-----------------|
| 从创意生成大纲/梗概（异步） | `POST /api/projects/generate-outline` | `{ idea }` → `jobId` | **已有**：`packages/backend/src/routes/projects.ts` |
| 查询大纲 Job | `GET /api/projects/outline-job/:jobId` | — | **已有** |
| 更新项目创意/梗概/风格 | `PUT /api/projects/:id` | `{ description?, synopsis?, visualStyle? }` | **部分已有**（以路由实现为准） |
| **生成第一集**文学剧本 | `POST /api/projects/:id/generate-first-episode` | `{ description?: string }` 或使用已保存创意 | **待实现**；能力可复用 `writeScriptFromIdea` / expand 与 Episode 1 写入 |
| **批量生成剩余集** | `POST /api/projects/:id/generate-all-episodes` | `{ targetEpisodes: number }` → `{ jobId }` | **待实现**；内部创建 `PipelineJob` 或独立队列 Job |
| **查询批量/解析进度** | `GET /api/pipeline/job/:jobId` | — | **已有**：`packages/backend/src/routes/pipeline.ts` |
| **解析剧本（制作准备）** | **`POST /api/projects/:id/parse`**（或 `:projectId/parse`，与路由统一） | 可选 `{ targetEpisodes? }` 等；→ `{ jobId }` | **待实现**；创建解析类 **PipelineJob**（或专用任务）；**不**在本接口语义内启动分镜/视频全链路 |
| **全量制作流水线（详情页）** | `POST /api/pipeline/execute` 或等价 | `projectId`、`targetEpisodes`、`visualStyle` 等 | **已有**；与大纲页 **「解析剧本」** 分属不同产品动作，勿混用文案 |

### 5.2 与 v4.1 实体字段的直接映射

| UI 区域 | 读写实体与字段 |
|---------|----------------|
| 故事创意 | `Project.description`（或约定扩展字段） |
| 故事梗概 | `Project.synopsis` |
| 视觉风格 | `Project.visualStyle` |
| 第一集 / 各集预览 | `Episode.rawScript`（DB 列 `script`），`episodeNum` 区分集数 |
| 总集数 | 配置常量 MVP=60 ↔ `targetEpisodes` / 项目元数据 |
| 异步任务 | `PipelineJob`、`PipelineStepResult` |

## 6. 与旧版「进入流水线」及全量 Pipeline 的关系

**v2.0** 起，大纲页主 CTA 为 **「解析剧本」**（见 **§3.6**），职责限于 **实体提取 + 形象槽位 + 分集概要**，**不**在本页触发生图、分镜、视频。

若产品仍需要「从项目一键跑完整制作 Pipeline」，应作为 **项目详情页** 的独立入口（如调用 **`POST /api/pipeline/execute`**），**不**与「解析剧本」共用同一文案或同一接口语义。

**补全缺失集**：用户点击 **「稍后再说」** 未批量生成时，**解析任务或服务端编排**应在解析前或解析流水线内补全缺失 **`Episode.rawScript`**（与 §3.3、executor 约定一致），避免「无剧本可解析」。

**全量 Pipeline 的智能起点（实现参考，非本页 CTA）**：当用户在**详情页**触发全量流水线时，若 **全部目标集数** 的 **`rawScript` 已齐**，执行器可 **跳过** 与「从 idea 拆多集」等价的步骤，从后续实体/场次步骤继续；若仍有缺集则先补集。步骤键以 `pipeline-executor` 与 Prisma 为准（如 `script-writing`、`episode-split`、`segment-extract`、`storyboard` 等）。

## 7. 关键交互规则小结

1. **进入页面**：优先保证 **第一集** 可预览（快速验证创意 → 剧本质量）。
2. **一键补全**：「生成剩余 N 集」批量写入 **`Episode.rawScript`**，异步可离页；**不要求**事先选择视觉风格。
3. **总集数**：MVP **固定 60**；与 **`targetEpisodes`** 对齐，后续可配置。
4. **「解析剧本」**：校验 **第一集 `rawScript` + 至少一项 `visualStyle`** → **`POST .../parse`** → 跳转 **`/project/:id`** 展示解析进度；后台完成 **§3.6** 所列职责。（风格与批量写集解耦。）
5. **「稍后再说」**：允许不批量生成；补集由 **解析前/解析内** 服务端编排处理（§3.3、§6）。
6. **精修路径**：本页以 **集级文学剧本** 为主；**场次级** 精修统一引导至 **项目详情 → 场次管理（Scene）**。
