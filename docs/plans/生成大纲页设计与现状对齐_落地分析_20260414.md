## 一、现状流水线（代码中的真实路径）

1. **项目列表** [`packages/frontend/src/views/Projects.vue`](../../packages/frontend/src/views/Projects.vue)  
   用户输入「想法」→ **`POST /api/projects/generate-outline`** → 返回 **`jobId`** → 跳转 **`/generate?jobId=...`**。

2. **生成页** [`packages/frontend/src/views/Generate.vue`](../../packages/frontend/src/views/Generate.vue)  
   - 无 **`jobId`** 则重定向回 **`/projects`**。  
   - 使用 **`pollOutlineJob`** 轮询 **`OutlineJob`**（[`OutlineJob` Prisma 模型](../../packages/backend/prisma/schema.prisma)），展示 **`result.outline`**（标题、摘要、metadata 等）。  
   - **并非**交互稿中的「第一集 `Episode.rawScript` 长文预览」。

3. **用户点击「开始」**  
   先 **`createProject`**（name / description）→ **`PUT /api/projects/:id`** 仅传 **`visualStyle`**。

4. **后端** [`packages/backend/src/routes/projects.ts`](../../packages/backend/src/routes/projects.ts) 中 **`PUT /:id`**  
   `prisma.project.update` 的 **`data` 仅包含 `name`、`description`**，**未**写入 **`synopsis`、`visualStyle`**。  
   因此前端传的 **`visualStyle` 当前实际上不会持久化**（实现缺陷，应在落地第 0 期修复）。

**结论**：现网是 **「OutlineJob 异步大纲 → `/generate` 展示大纲 → 用户确认后创建 Project → 进入项目 Pipeline」**；两份设计描述的主线是 **「已有 Project：梗概 + 第一集剧本 + 可选批量集 + 风格 → 进 Pipeline」**。  
**已拍板**（见 §九）：主路径 **强制 `projectId`**，与现网 **`jobId` 先入页** 不一致，**入口与后端编排须改造**（见 §九）。

## 二、交互设计 vs 现有能力

| 交互设计要点 | 现状 | 差距 |
|--------------|------|------|
| 路由 **`/generate`** | 已注册 | 一致 |
| **`?projectId=`** 续写已有项目 | 仅支持 **`?jobId=`** | 需扩展查询参数与数据加载 |
| 双栏：创意 / 梗概、第一集预览、批量剩余集、多选风格、保存草稿、Monaco | 单页大纲卡片 + 单选风格 +「开始」建项 | UI 与状态机需迭代或分阶段交付 |
| 批量进度、离页恢复 | 仅 Outline 轮询 | 依赖 **Project 维度 + PipelineJob**（或专用 Job） |
| **`PUT` 保存梗概/风格** | `PUT` 未接 `synopsis` / `visualStyle` | **必须先补 API** |

## 三、API 设计 vs 现有能力

| API 设计 | 现状 | 落地建议 |
|----------|------|----------|
| `POST .../episodes/generate-first` | 无等价端点；存在 **`POST .../episodes/:id/expand`** 与 **`generate-outline`（无 project）** | 新建或合并能力，并明确与 Outline 的先后顺序 |
| `POST .../episodes/generate-remaining` | 无；[`POST /api/pipeline/execute`](../../packages/backend/src/routes/pipeline.ts) 为 **idea 冷启动整条流水线** | 新建 **仅批量写 `Episode.rawScript`** 的异步任务，与 `execute` 解耦 |
| 进度 **`GET /api/pipeline-jobs/:jobId`** | 实际为 **`GET /api/pipeline/job/:jobId`**，且响应常为 **`{ success, data }`** | **以现路径为准**扩展字段；避免再造一套路径 |
| **`Project.storyContext`** | Prisma 中 **不存在** | 迁移新增 |
| **`PipelineJob` 区分 script-batch** | 无 **`type` / `jobType`** 列 | 迁移新增推荐；MVP 也可用 **`currentStep` 字符串约定**（可维护性较差） |
| 细粒度进度 `current/total` | `PipelineJob.progress` 为 **0–100 整数** | 使用 **`Json?` 扩展字段**（如 `progressMeta`）或写入 **`PipelineStepResult.output`** |

## 四、两条设计文档之间的关系

- **交互稿**：定义页面结构、状态与入口（以 **`/generate`**、`projectId` 为主）。  
- **API 稿**：定义「服务端攒上下文、极简入参」及 **`storyContext`、批量循环** 等。  

二者匹配，但均依赖 **尚未实现** 的模型字段与新路由；且都假设 **Project 已存在** 或可稳定关联，与现网 **先 OutlineJob、后建 Project** 不同。

**入口策略（已拍板）**

- **方案 B**：列表「快速创建」等入口须 **先 `POST /projects`（或等价）**，再 **`/generate?projectId=`**；大纲/第一集生成与 `OutlineJob` 的衔接方式在实现中定（可 `generate-outline` 带 `projectId`，或并入 **`generate-first`**）。  
- **方案 A（仅 `jobId`）**：**不作为与后续功能等价的主路径**；见 §9 第 8 条。

## 五、分阶段落地方案

### 第 0 期：修正与契约对齐（小改动、优先）

1. **修复** `PUT /api/projects/:id`：支持 **`synopsis`、`visualStyle`**（与 [`Project`](../../packages/backend/prisma/schema.prisma) 字段一致），并验证 [`Generate.vue`](../../packages/frontend/src/views/Generate.vue) 风格能落库。  
2. **文档与实现统一**：进度轮询以 **`GET /api/pipeline/job/:jobId`** 为准；扩展字段时保留 **`{ success, data }`** 兼容。  
3. **路由注册**：新增 `POST` 静态路径须避免被 **`/:id`** 吞掉（与现有 `generate-outline`、`outline-job` 排布原则相同）。

### 第 1 期：数据模型（支撑 API 稿 §3）

1. 迁移：**`Project.storyContext`**（`TEXT`，可空）。  
2. 迁移（推荐）：**`PipelineJob.jobType`**（如 `full-pipeline` | `script-batch`）。  
3. （可选）**`PipelineJob.progressMeta`** `Json?`，存 `{ current, total, message }`。

### 第 2 期：生成大纲页 2.0（对齐交互主路径）

1. **实现** `POST /api/projects/:projectId/episodes/generate-first`：校验归属 → AI → 写 **`Project.synopsis`**、**Episode 1 `rawScript`**、初始化 **`storyContext`**（逻辑见 API 设计文档）。  
2. **前端**：`Generate.vue` 支持 **`projectId`**；双栏创意/梗概、第一集预览（从 `rawScript` 渲染）、多选 **`visualStyle`** + **`PUT` 保存**。

### 第 3 期：批量剩余集 + 「解析剧本」+ 详情页全量 Pipeline（与交互 v2.0 对齐）

1. **实现** `POST .../episodes/generate-remaining`：创建 **`PipelineJob`（`jobType=script-batch`）**，后台逐集续写 **`Episode.rawScript`**；注意 **`storyContext` 截断** 与 **Token 上限**（实现见 [`project-script-jobs.ts`](../../packages/backend/src/services/project-script-jobs.ts)）。  
2. **扩展** `GET /api/pipeline/job/:jobId`：响应增加 **`jobType`、`progressMeta`**（`{ current, total, message }`），保留 **`{ success, data }`**。  
3. **大纲页主出口「解析剧本」**：**`POST /api/projects/:id/parse`** → **`jobType=parse-script`**，职责为实体提取 + 形象槽位 + 分集概要；**不**替代详情页 **`POST /api/pipeline/execute`**。  
4. **详情页全量流水线**：若 **`1..targetEpisodes`** 均已具备有效 **`rawScript`**，[`pipeline-executor`](../../packages/backend/src/services/pipeline-executor.ts) **`executePipelineJob`** 内 **跳过 script-writing / episode-split**（与交互稿 §6 一致）。

### 第 4 期（体验增强）

- 全部剧本预览、Monaco 高级编辑、保存草稿、离页恢复（可复用或扩展 **`GET /api/pipeline/status/:projectId`** 查询「进行中的 script-batch」）。

## 六、风险与依赖

1. **OutlineJob 与 Project**：在 **强制 `projectId`** 前提下，须约定 **OutlineJob 是否仍独立存在**、或与 **`projectId` 绑定**；梗概/第一集落库时机见 §9 第 2 条，避免与 **`generate-first`** 重复写入。  
2. **`storyContext` 体积**：需 **上限 + 滑动窗口**，避免 prompt 无限膨胀。  
3. **并发**：同一 **`projectId`** 上避免重叠两个 script-batch（唯一 running Job 或锁）。  
4. **失败续跑**：从 **`max(episodeNum)+1`** 继续；MVP 可采用 **「单集成功后再 `episode.create`」**，避免半集脏数据。  
5. **现网 `PUT` 缺陷**：未修前，**任何依赖 `visualStyle` 的流水线/场次默认值都可能不准**。

---

## 七、小结

- 两份设计**方向正确**，且 API 文档中「与仓库现状差异」已与真实代码对齐。  
- **最大断层**仍是：**现网 OutlineJob + `jobId` 入页** vs **已拍板：Project 中心 + 强制 `projectId`**。落地顺序建议不变：**先修 `PUT` 与 `storyContext` 等 schema，再改入口与 `Generate.vue`，再上新 POST / 批量 Job，最后改 executor**。

## 八、已确认决策（2026-04-14）

| 序号 | 主题 | 结论 |
|------|------|------|
| 1 | 主入口 | **方案 B**：以 **`/generate?projectId=`** 为主路径；进入生成大纲页前须已有 **`Project`**。 |
| 2 | OutlineJob 结果与 `Project` / `Episode` | **按选项一理解**：在用户确认大纲并触发**业务落库**（实现中对应原「点开始」前后之一，以 PR 为准）时，即写入 **`Project.synopsis`** 并保证 **`Episode` 第 1 集 `rawScript`** 进入业务表，**不长期仅依赖 `OutlineJob.result`**。若「2. 1」本义不同，请勘误本条。 |
| 3 | `PUT /api/projects/:id` | **是**：第 0 期必须支持 **`synopsis`、`visualStyle`**。 |
| 4 | `Project.storyContext` | **是**：迁移新增该字段（见 API 设计稿 §3.1）。 |
| 5–7 | `jobType`、进度接口形态、`execute` 扩展 | **待定**（另有改动后再记入本文与 API 稿）。 |
| 8 | `?jobId=` 兼容 | **强制先 `projectId`**：不以「仅 `jobId`」作为与后续功能等价的主路径；迁移期可短暂 **重定向或提示** 至带 `projectId` 的流程，直至入口改造完成。 |
