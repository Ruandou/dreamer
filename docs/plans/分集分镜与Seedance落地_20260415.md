# 分集分镜编辑与 Seedance 自动参考图 — 落地计划（含验证示例）

**版本**：与讨论稿对齐（2026-04-15）  
**数据模型**：[packages/backend/prisma/schema.prisma](packages/backend/prisma/schema.prisma)（Scene / Shot / CharacterShot / Location / CharacterImage / SceneDialogue / Take）

---

## 1. 目标摘要

- **分集管理**：聚合接口返回场次 + 定场图 + 多镜 + CharacterShot + 台词，供 Shot 卡片展示与编辑。
- **生成**：从同一套 DB 记录组装 Seedance 2.0 所需 **prompt（含「图片 n」与时间轴，§4）** 与 **参考图 URL 列表**；与现有队列 [`video.ts`](packages/backend/src/queues/video.ts)、[`SceneService.enqueueVideoGenerate`](packages/backend/src/services/scene-service.ts) 衔接。

---

## 2. 现状与差距（简）

| 能力 | 现状 |
|------|------|
| 聚合列表 | `GET /api/scenes?episodeId=` 仅含 `takes`，缺 `location` / `shots` / `characterShots` / `dialogues` |
| CharacterShot API | 无独立 PATCH/POST，仅流水线内写入 |
| 自动生成参考图 | `enqueueVideoGenerate` 未从 DB 组装 `imageUrls`；prompt 仅 [`stitchScenePrompt`](packages/backend/src/services/scene-prompt.ts) |
| 分镜脚本任务 | [`applyScriptContentToEpisode`](packages/backend/src/services/episode-service.ts) 每场单镜，无多 Shot / CharacterShot / SceneDialogue 持久化 |

## 3. 验证示例：从 Prisma 记录到 Seedance 语义（宋应星破旧屋内）

以下示例用于验收 **「分镜编辑可视化」** 与 **「自动应用图片」** 的规则是否一致；ID 可为示意。

### 3.1 数据存储假设

- **Project**：`visualStyle` 含 `realistic`, `cinematic`, `vintage`, `ancient_male` 等（生成结尾文案可合并 Scene + Project，见下）。
- **Location `loc_001`**：`imageUrl` = 定场图。
- **Character**：宋应星 `char_song`，母亲 `char_mother`。
- **CharacterImage**：各角色 `type: base` 的 `avatarUrl` 已填。
- **Scene `scene_001`**：`locationId`, `visualStyle`（如 `["realistic","cinematic","vintage"]`），`duration` 与两镜总时长一致（例：10000 ms）。
- **Shot**：`shot_1` order=1，`shot_2` order=2；`description` 为长文案（含剧情与台词原文）。
- **CharacterShot**：
  - `shot_1` → 仅 `img_song_base`，`action` 摘要动作；
  - `shot_2` → `img_mother_base` + `img_song_base`（两条），`action` 各不同。
- **SceneDialogue**：两条，含 `startTimeMs` / `durationMs` / `text` / `voiceConfig`（结构化）。

### 3.2 标签 `@CharacterN` 与 references 顺序规则（与示例一致）

1. **参考图顺序（语义层 `references[]` / HTTP 层 `content[].image_url`）**  
   - 第 1 张：**场景** — `Location.imageUrl`（产品层可记 `role: location`；方舟侧仍为 **`image_url` + `role: "reference_image"`**，见 §4.1）。  
   - 后续：**角色** — 按「**镜头 order 升序** → 每个镜头内 **CharacterShot 顺序**」遍历，**每个 `characterId` 首次出现**时分配序号 **图片1、图片2…**（仅 `image_url` 计数），并把对应 `avatarUrl` 加入列表。

   对本例：`image_url` 顺序为 **[定场图, song_base, mother_base]** → 提示词中依次为 **图片1**（环境）、**图片2**（宋应星形象）、**图片3**（母亲形象）；人名替换须与序号一致（§4.2）。  
   **文案替换**：讨论稿可用 `@CharacterN` 作内部占位；**调用方舟前统一为「图片 n」**。

2. **Prompt 内人名替换**  
   - 将剧本/镜头描述中的 **角色名** 替换为 **「图片 n」**（与 §3.2 第 1 条中 URL 顺序一致）；需 **正则转义**、并建议 **按名字长度降序** 替换，避免短名误伤长名。  
   - 口语/台词中昵称（如「应星」「娘」）是否替换由产品定；本示例保留台词内的「应星」等，仅替换用于 **指代角色本体** 的完整名。

3. **镜头拼接**  
   - 与现有 [`stitchScenePrompt`](packages/backend/src/services/scene-prompt.ts) 一致：`[Shot 1] … [Cut to] [Shot 2] …`（或统一为 `[景别, 运镜]` 前缀，二选一并全仓统一）。

4. **对话时间轴（MVP）**  
   - 从 `SceneDialogue` 按时间排序生成段落，例如（角色对应 **图片2 / 图片3**）：  
     `3-5秒：图片2用…声说道："…"`  
     `7.5-10秒：图片3用…声说道："…"`  
   - 音色文案由 `voiceConfig`（gender/age/timbre 等）拼成可读短句；与独立 **TTS / audio 参数** 可后续再接，不阻塞 MVP。

5. **结尾画风句**  
   - 用 `Scene.visualStyle`（必要时合并 `Project.visualStyle`）拼成一句中文说明 +「8K超高清」等，与示例「真人写实，电影风格，复古调色」同类。

6. **时长**  
   - `duration`（秒）与 `Scene.duration` 或各 `Shot.duration` 之和一致；示例为 10s。

### 3.3 期望的 prompt / references（摘录）

- **Prompt**：在拼接后的镜头文案中，将角色名替换为与方舟约定一致的引用符（见 §4.3：**官方推荐「图片 n」**，讨论稿中的 `@CharacterN` 为早期占位，落地时应与文档对齐）；另起「对话时间轴」块；最后为画风句。  
- **references（语义）**：见上「location + 两角色图」三条；**HTTP 层**为 `content[]` 中多条 `image_url`，且官方示例带 **`role: "reference_image"`**（见 §4.1）。

## 4. 与仓库内 Seedance 实现的映射（重要）

### 4.1 方舟官方《Seedance 2.0 系列教程》核对（2026-04-15）

**文档**：火山方舟 → 视频生成 → **Seedance 2.0 系列教程**（与「创建视频生成任务 API」配套；SDK 示例为 `client.content_generation.tasks.create`，对应 REST **`/api/v3`** 下内容生成任务）。

**请求体（与官方 Python 示例一致）**：

- **`model`**：`doubao-seedance-2-0-260128` 或 **`doubao-seedance-2-0-fast-260128`**（与仓库 [`SEEDANCE_MODEL`](packages/backend/src/services/ai/seedance.ts) 选用一致即可）。
- **`content`**：**数组**，首段一般为 `{"type":"text","text":"..."}`，其后可接多张图、多段视频、多段音频等。
- **参考图片**：`{"type":"image_url","image_url":{"url":"..."},"role":"reference_image"}` —— **官方示例中每张图片均带 `role: "reference_image"`**；当前仓库 [`buildArkRequest`](packages/backend/src/services/ai/seedance.ts) **未**附带 `role`，**落地时应补上**，与文档一致。
- **多模态配额（教程原文）**：文本 + **图片 0～9 张** + 视频 0～3 个 + 音频 0～3 个；不支持「纯文本+纯音频」等无视频路径的组合（以文档为准）。
- **顶层参数**：`duration`、`ratio`（如 `16:9`、`9:16`、`adaptive`）、`generate_audio`、`watermark` 等与教程一致；输出分辨率 **480p / 720p**，时长 **4～15 秒**（教程表格）。

**不存在**独立 JSON 字段 **`content.references`**；多参考图 = **`content` 数组中多条 `image_url`**。

### 4.2 提示词规范：「图片 n」与产品层标签（必读）

**官方「提示词技巧」原文要点**：

- 提示词中须使用 **「素材类型 + 序号」** 引用素材。
- **「图片 n」** 表示：在 `content` 数组中，第 **n** 个 **`type=image_url`** 的条目（**仅按图片类计数，从 1 开始**）。
- 同理有 **视频 n**、**音频 n**；**不支持用 Asset ID 在文案里指代素材**。

**对 Dreamer 产品映射**：

- 若定场图为 **第 1 个** `image_url`，则文案中环境锚点应写 **「图片1」**（或教程示例中的「首帧为图片1」等），而非 `@Character1`。
- 角色形象依次为 **图片2、图片3**…（按 `image_url` 在同类型中的顺序）。  
- 讨论稿里的 **`@CharacterN`** 与官方 **「图片 n」** 不同；**落地时 Builder 应输出「图片 n」话术**（或与内部 `@CharacterN` 二选一并在 `buildArkRequest` 前统一替换为「图片 n」），以符合官方 Seedance 2.0 提示词指南。

### 4.3 合规：真人脸参考图（教程「便利创作」）

- **不支持**直接上传含**真人脸**的参考图/视频（除非虚拟人像、已授权入库 `asset://`、或符合规则的模型产物二创等）。
- 落地「角色定妆图」时：需评估是否走 **虚拟人像 / 授权素材 / 非写实风格**，或依赖 **本账号 Seedance 产物的二次创作** 等路径；否则可能触发审核失败。

**本条款**写入验收与运维说明，避免仅依赖「公网 URL 定妆图」。

### 4.4 仓库实现与待对齐项

当前 [`buildArkRequest`](packages/backend/src/services/ai/seedance.ts)：

- 使用 **`content`** 数组：`text` + 多个 **`image_url`**（无 `role`）；并在 **text** 中拼接 `--ratio`、`--fps`、`--dur`（与 SDK 示例「顶层 `ratio`/`duration`」并存时需确认是否重复，**建议以官方文档为准做一次收敛**，避免参数冲突）。

**待对齐**：

1. 每个 **`image_url` 块**增加 **`role: "reference_image"`**（若仅一张场景+多张角色，均为 `reference_image`，序号由「图片 n」在文案中区分）。  
2. **Prompt** 中角色/场景引用改为 **「图片1」「图片2」…**（与 §4.2）。  
3. **轮询状态**：教程示例为 `succeeded`；仓库 [`mapStatus`](packages/backend/src/services/ai/seedance.ts) 已映射 `succeeded` → `completed`，**可保持不变**。  
4. **合规**：产品侧提示用户真人脸素材限制；技术侧可选对接虚拟人像 / 资产 ID 流程（后续迭代）。

## 5. 分镜脚本编辑器交互（编辑框 / 编辑保护 / 人物实体 / @ 选形象）

**目标（产品）**

- 分镜脚本以**可编辑区域**承载（非整页只读），必要时与「梗概」等字段区分布局。
- **编辑保护（简单方案）**：**默认不可编辑**（只读展示）；一个 **「编辑」** 按钮切到可编辑态；可再配合「取消/保存」或离开页脏提示。不做复杂权限矩阵。
- **角色呈现**：脚本中出现的角色在界面层以**人物实体**展示：**头像缩略图 + 姓名**，使用统一 chip/ token 样式与正文区分；底层仍须能映射到 `Character` / `CharacterImage`，供生成与 Seedance 参考图顺序对齐。
- **`@` 唤起选型**：编辑态下输入 **`@`** 弹出**角色形象选择器**（项目内角色列表 + 各角色下可选形象）；选中后插入对应实体（绑定 `characterId` + `characterImageId`），避免仅靠纯文本人名。

**实现要点**

- **存储与解析**：在 Episode 现有脚本字段（以 [`schema.prisma`](../../packages/backend/prisma/schema.prisma) 为准）上明确序列化格式：推荐 **JSON 文档**（Tiptap / ProseMirror 等）存放带 inline「角色节点」的结构化内容；若短期保留纯文本，须定义与 AI 产物的**导入/升格**策略，避免无法稳定还原实体与 `@` 插入语义。
- **编辑器**：Inline「角色」作为自定义节点或嵌入式组件；`@` 触发 `mention`/`combobox` 下拉，数据源为角色 + `GET /api/characters/:characterId/images`（或项目级聚合接口减少请求）。
- **与后端流水线**：`applyScriptContentToEpisode`（或后续 parser）从文档结构或约定 DSL 提取角色引用，再写入 `Shot` / `CharacterShot`；prompt 侧仍按 §3、§4「图片 n」规则生成，与前端 chip 顺序无关但须**可验证一致**。

**任务补充**

- **P1**：分镜脚本编辑区 UI + 实体组件 + `@` 弹窗 + **默认只读 +「编辑」按钮进入编辑**（二态即可）。
- **P1**：与保存接口对齐（单次 PUT/PATCH episode.script 或分段子资源），并补充前后端测试。

## 6. 实施任务清单（与先前计划一致）

1. `GET /api/episodes/:episodeId/scenes` 聚合查询（含 location、shots、characterShots、characterImage、character、dialogues、takes）。  
2. `PATCH /api/character-shots/:id`、`POST /api/shots/:shotId/character-shots` + 权限校验。  
3. 实现 `buildSeedanceScenePayload(sceneId)`（或等价命名），并在 **`POST /api/scenes/:id/generate` 未传 `imageUrls` 时** 自动注入；prompt 含 **「图片1」…** 引用与时间轴（§4.2）；并更新 [`seedance.ts`](packages/backend/src/services/ai/seedance.ts) **`image_url` 项附带 `role: "reference_image"`**，必要时收敛 text 内 `--dur` 与顶层 `duration` 重复问题（§4.4）。  
4. 分镜 AI 与 `applyScriptContentToEpisode`：多 Shot、`imageName` → CharacterImage、`SceneDialogue` 写入。  
5. 前端分集 Tab：Shot 卡片、切换形象、添加角色、生成走自动参考图。  
6. 测试：以 **§3 示例** 作固定夹具数据，断言 prompt 含 **「图片 n」**、URL 顺序、`imageUrls.length` ≤ 9；单测覆盖 `buildArkRequest` 的 `role` 字段。  
7. 提示词质量：关键路径可对照 [官方提示词指南](https://www.volcengine.com/docs/82379/2222480?lang=zh) 与 [`docs/skills/reference.md`](docs/skills/reference.md) 词库做人工抽检（非阻塞自动化）。
