# 角色与场地提取及图片提示词全面优化计划

## 问题分析

### 当前问题

1. **模型输入/输出过大导致失败**: `script_visual_enrichment` 批量处理所有角色和场地的提示词，当项目有大量角色/场地时，JSON 输出可能超过模型处理能力
2. **角色形象提示词过于笼统**: 当前仅描述"外貌、发型、服装"等，缺少具体面部特征（脸型、眼睛、鼻子、耳朵、眉毛等），导致不同角色的形象缺乏区分度
3. **场地提示词优化空间**: 定场图提示词可以更具体，增加空间层次感和细节描写

### 核心瓶颈

- **批量一次性处理**: 所有角色 + 所有场地 → 单个 AI 调用 → 输出巨大 JSON
- **Prompt 长度限制**: 当前 `maxOutputTokens: 4096`，但多角色多场地时容易超限
- **面部特征缺失**: 角色形象缺乏辨识度，无法确保"同一人"在不同镜头中的一致性

---

## 优化方案

### 1. 性能优化：分批处理策略

#### 方案 A：拆分视觉增强为两个独立步骤

**优点**: 降低单次调用负载，失败时影响范围小
**缺点**: 增加一次 API 调用

**实现**:

```
原流程: script_visual_enrichment (角色 + 场地 → 单次调用)
新流程:
  - location_visual_enrichment (仅场地 → 调用1)
  - character_visual_enrichment (仅角色 → 调用2)
```

#### 方案 B：角色分批处理（推荐）

**优点**: 超大型项目可进一步拆分
**实现**:

```typescript
// 每批最多处理 5 个角色
const CHARACTER_BATCH_SIZE = 5
for (let i = 0; i < characters.length; i += CHARACTER_BATCH_SIZE) {
  const batch = characters.slice(i, i + CHARACTER_BATCH_SIZE)
  await processCharacterBatch(batch)
}
```

#### 方案 C：并行化（需评估质量影响）

**注意**: 根据用户偏好，并行化可能影响一致性，需先分析

**潜在问题**:

- 不同批次间角色风格可能不一致
- 场地视觉风格可能不统一
- **需要共享上下文**（如项目视觉风格、剧本梗概）

**建议**: 暂不实施并行化，优先保证质量

### 2. 角色形象提示词优化：面部特征结构化

#### 当前问题

```
现状: "年轻男子，黑色短发，穿着蓝色衬衫"
问题: 缺乏区分度，不同角色可能生成相似形象
```

#### 优化方案：引入面部特征模板

**新增结构化的面部特征描述**:

```typescript
interface FacialFeatures {
  // 脸型
  faceShape: '圆形脸' | '方形脸' | '长形脸' | '心形脸' | '菱形脸' | '椭圆形脸'

  // 眼睛
  eyeShape: '杏眼' | '桃花眼' | '丹凤眼' | '圆眼' | '细长眼'
  eyeSize: '大' | '中等' | '小'
  eyeColor?: string // 特殊颜色

  // 眉毛
  eyebrowShape: '剑眉' | '柳叶眉' | '平眉' | '拱形眉' | '浓眉'

  // 鼻子
  noseShape: '高挺鼻梁' | '圆润鼻头' | '鹰钩鼻' | '小翘鼻' | '直鼻'

  // 嘴唇
  lipShape: '薄唇' | '厚唇' | 'M字唇' | '微笑唇'

  // 耳朵
  earShape?: '招风耳' | '贴面耳' | '正常'

  // 特殊标记
  distinctiveMarks?: string[] // 如：左眼下方有泪痣、右眉有疤痕
}
```

**优化后的 Prompt 模板**（以 base 为例）:

```
【基础定妆 - 正面全身照】
站立于纯色影棚背景（中灰色）；正面全身构图（头顶到脚底完整可见）；

面部特征（须具体、可区分）:
- 脸型：{faceShape}
- 眼睛：{eyeSize}{eyeShape}，目光{锐利/温和/深邃}
- 眉毛：{eyebrowShape}
- 鼻子：{noseShape}
- 嘴唇：{lipShape}
- 特殊标记：{distinctiveMarks}

整体外貌：{age}岁{gender}，气质{气质描述}
发型：{hairStyle}，发色{hairColor}
服装：{clothing}
姿态：正面站立，自然表情

构图与背景：正面全身，纯色中灰背景，电影级画质
```

**优势**:

1. 每个角色有独特的面部"指纹"
2. 换装形象可明确"保持不变"的面部特征
3. AI 绘图时有更具体的锚点，提升一致性
4. 正面全身照确保角色完整可见，便于视频生成时全身动作

## 形象槽位策略

### 保留

- **base（基础定妆）**: 正面全身照，纯色背景，包含详细面部特征
- **outfit（换装）**: 正面全身照，保持面部特征不变，仅更换服装

### 移除

- ~~**expression（表情）**~~: 不需要，表情变化由分镜动作描述控制
- ~~**pose（姿态）**~~: 不需要，姿态变化由分镜动作描述控制

### 原因

1. 表情和姿态是动态的，应在视频分镜中通过动作指令实现
2. 固定形象图只需提供"角色长什么样"的参考
3. 减少不必要的 AI 调用，降低成本
4. 简化用户界面，避免形象管理复杂化

### 3. 场地提示词优化：空间层次与细节

#### 当前问题

```
现状: "咖啡厅内景，暖光，木质桌椅"
问题: 缺乏空间深度和具体细节
```

#### 优化方案：四段式增强

**增强的四段式结构**:

```
1. 空间与环境（增强版）:
   - 建筑结构：层高、窗户类型、天花板特征
   - 材质：墙面（砖/漆/木板）、地面（瓷砖/木地板）、家具材质
   - 标志性物体：吧台、吊灯、装饰画、书架
   - 空间层次：前景、中景、背景元素

2. 光影与氛围（增强版）:
   - 光源：自然光（窗户方向）+ 人工光（灯具类型）
   - 光线方向：侧光、逆光、顶光
   - 色温：暖光（2700K）、中性光（4000K）、冷光（6000K）
   - 氛围词：温馨、压抑、神秘、空旷

3. 构图与视角（增强版）:
   - 景别：广角全景（展现空间）、中景（聚焦区域）、特写（细节）
   - 角度：平视、俯视、仰视
   - 引导线：走廊纵深、桌面延伸、光线轨迹

4. 风格与画质:
   - 融入项目视觉风格
   - 画质词：电影质感、8K超高清、浅景深、 volumetric lighting
```

**示例**:

```
优化前: "咖啡厅内景，暖光，木质桌椅"

优化后: "挑高3米的复古咖啡厅空间。红砖墙面搭配深色原木横梁，天花板悬挂三盏黄铜工业风吊灯，地面为做旧木地板。前景是吧台区域，中景可见两排皮质沙发卡座，背景是整面落地窗与窗外街景。下午45度侧光透过百叶窗投射条纹光影，与吊灯的2700K暖光交织，营造温馨慵懒的氛围。广角全景平视构图，引导线从吧台延伸至落地窗，展现空间纵深。电影质感，8K超高清，浅景深突出中景沙发区，volumetric lighting增强光柱效果。"
```

## 实施步骤

### Phase 1: 性能优化（拆分批量处理）

1. 将 `applyScriptVisualEnrichment` 拆分为场地和角色两个独立调用
2. 添加分批处理逻辑（角色每批最多5个）
3. 增加详细日志记录每批处理状态
4. 更新 pipeline 步骤状态显示

**预估工作量**: 2-3小时

### Phase 2: 角色提示词结构化

1. 更新 `CHARACTER_BASE_PROMPT_TEMPLATE`，改为正面全身照，增加面部特征结构
2. 修改 `buildScriptVisualEnrichmentUserContent`，在角色输入中添加面部特征引导
3. 更新 `generateCharacterSlotImagePrompt` 相关模板，移除 expression/pose 模板
4. 修改 `VISUAL_ENRICHMENT_TEMPLATE` 中的角色规则，仅保留 base 和 outfit
5. 更新 `script-visual-enrich.ts` 中的类型定义，移除 expression/pose 处理逻辑
6. 添加示例数据测试

**预估工作量**: 3-4小时

### Phase 3: 场地提示词增强

1. 更新 `VISUAL_ENRICHMENT_TEMPLATE` 中的场地规则
2. 增强 `LOCATION_ESTABLISHING_TEMPLATE` 四段式描述
3. 添加空间层次、材质、光影引导词
4. 测试不同场地类型（室内/室外/日/夜）

**预估工作量**: 2-3小时

### Phase 4: 测试与验证

1. 编写单元测试覆盖新模板
2. 使用实际项目测试端到端流程
3. 对比优化前后的图片质量
4. 检查模型调用日志，确认无超时/错误

**预估工作量**: 2小时

## 风险与注意事项

1. **模型输出格式稳定性**: 结构化面部特征可能增加 JSON 输出复杂度，需加强解析韧性
2. **提示词长度控制**: 新增细节描述须控制在 120 字以内（base）/ 100 字以内（衍生）
3. **向后兼容**: 现有项目的角色/场地须能正常迁移，不破坏已有数据
4. **API 成本**: 拆分调用可能增加 1 次 API 请求，成本增加约 ¥0.01-0.02/项目
