# 图片生成服务（独立模块）— 阶段范围

**结论（按当前共识）**：

1. **方舟调用与转存**仍集中在 [`image-generation.ts`](../../packages/backend/src/services/image-generation.ts)，保持可单测、无 HTTP 耦合。
2. **实际执行走异步 Job**（与现有视频生成一致：BullMQ + Redis + Worker），避免长耗时阻塞请求、可重试、便于后续接 SSE/任务列表。
3. **对外的 HTTP、DeepSeek 提示词、前端** 仍可整体后做；本阶段先把 **服务 + 队列 + Worker** 闭环。

---

## 阶段一：服务层 + Job 执行（优先落地）

### 1. 纯服务（已有或小幅补齐）

**职责**：封装火山方舟 `POST /images/generations`：

- 文生图、图生图/指令编辑（参考图默认 Base64 data URL，避免本地 URL 对方舟不可达）
- `*AndPersist`：拉取方舟临时 URL → 写入对象存储 → 返回持久 URL
- 环境变量：`ARK_API_KEY`、`ARK_API_URL`、`ARK_IMAGE_T2I_MODEL`、`ARK_IMAGE_EDIT_MODEL`、`ARK_IMAGE_EDIT_USE_BASE64` 等

**测试**：仍以 **纯函数/可 mock 单元测试** 为主（如 `strengthToGuidanceScale`、请求体组装若拆出）；对 `fetch` 可 mock 或只做集成脚本联调。

### 2. Job 执行（本阶段与 1 一起做）

**对齐现有模式**（参考 [`queues/video.ts`](../../packages/backend/src/queues/video.ts)、[`worker.ts`](../../packages/backend/src/worker.ts)）：

- 新增队列：例如 `image-generation`（名称以最终实现为准）
- **JobData**（建议放在 `packages/shared` 类型）：区分 `kind`（如 `text_to_image` / `image_edit`）、prompt、可选 `referenceImageUrl`、`strength`、以及 **业务回写所需 id**（如 `characterImageId` / `locationId`，便于 worker 内更新 Prisma；若阶段一暂不连库，可先只 `console`/返回结果，但建议一次性设计好 payload，避免二次改造）
- **Worker**：`job` 处理器内 **只调用** `image-generation` 的 `generateTextToImageAndPersist` / `generateImageEditAndPersist`，成功后再更新对应表字段或写入任务结果
- **Redis**：沿用 `REDIS_URL`，与 video/import 共用连接方式

**推荐数据流**：

```mermaid
flowchart LR
  enqueue [enqueue 调用方 后续再接 HTTP]
  Q[BullMQ image queue]
  W[Worker]
  S[image-generation service]
  Store[storage]
  DB[(Prisma 可选回写)]
  enqueue --> Q --> W --> S --> Store
  W --> DB
```

**说明**：

- **「用 Job 执行」**指：任何正式调用路径最终应是 `queue.add` → Worker，而不是在路由里直接 `await generate...`（长耗时 + 难重试）。
- **任务中心 / `/jobs` 展示**：属于产品层，可仍列为阶段二；但 Job 落库或 `ModelApiCall` 记录若与现有任务列表模型一致，可在实现 Worker 时 **预留** 字段，避免以后再改一遍。

## 阶段二及以后（先等一等）

- 用户触发的 HTTP API（创建任务、查询状态）
- DeepSeek 自动生成英文提示词

- Kling 备选 Provider

（若仓库里已有同步路由调用 `image-generation`，后续应改为 **仅入队**，由 Worker 执行。）

---

## 验收（阶段一）

1. `image-generation` 服务在本地或测试中能独立完成方舟调用 + 转存（有 key 时联调）。
2. `pnpm dev` / `worker` 进程启动后，能 **消费一条测试 Job** 并成功跑完（可暂时用手动 `queue.add` 脚本或最小集成测试，不强制先写 HTTP）。
3. Redis 未启动时行为明确（与现有队列一致：文档或启动说明中注明）。
4. 单元测试覆盖服务层关键逻辑；Worker 可对处理器做 mock 级测试（可选）。
